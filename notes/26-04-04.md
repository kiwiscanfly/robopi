## Raspberry Pi boot troubleshooting

### Red light, no green activity
- Solid red = powered but not booting
- No green flicker = SD card not being read — reseat the card first
- Green flicker then hang = OS started loading but crashed, likely a corrupted filesystem

### Checking SD card on Mac
macOS cannot verify ext4 (the Linux root partition). Only the FAT32 boot partition can be checked:

```bash
diskutil verifyVolume /dev/disk17s1   # boot (FAT32) — works
diskutil verifyVolume /dev/disk17s2   # root (ext4) — will error, this is expected
```

If the boot partition is clean but the Pi still won't boot:
- suspected the root filesystem was likely corrupt — re-flashed still nothing
- suspected the microSD card, tried another still nothing
- realised I can hook up a display
- turns out the WiFi was busted on that Pi after trying it on another

### Re-flashing
Use Raspberry Pi Imager. In the settings gear before flashing, configure:
- Hostname: `robopi`
- Username/password
- SSH enabled
- WiFi credentials

## Additional WiFi / NetworkManager issues

### Symptoms
- `nmcli device wifi connect` fails with: `802-11-wireless-security.key-mgmt: property is missing`
- `nmtui` keeps showing the password dialog even with the correct password
- `raspi-config` WiFi option errors: "There was an error running option S1 Wireless LAN"

### Root cause
A corrupted NetworkManager connection profile created by Raspberry Pi Imager during flashing.

### Fix
Delete the bad profile and reconnect from scratch:

```bash
sudo nmcli con delete "NetworkName"
sudo nmcli dev wifi connect "NetworkName" password "password" ifname wlan0
```

### Verify autoconnect is enabled
```bash
nmcli con show "NetworkName" | grep autoconnect
# connection.autoconnect should be: yes
```

### Check IP address


## Resistor colour codes

4-band resistors: first two bands = digits, third = multiplier, fourth (gold/silver) = tolerance.

| Colour | Digit | Multiplier |
|--------|-------|------------|
| Black  | 0     | ×1         |
| Brown  | 1     | ×10        |
| Yellow | 4     | ×10,000    |
| Purple | 7     | —          |
| Gold   | —     | ±5% tol.   |

- Brown Black Black Gold = **10Ω** — too low for an LED on 3.3V GPIO
- Yellow Purple Black Black Gold = **470Ω** — good for LED on 3.3V GPIO

### Choosing a resistor for an LED
```
R = (supply voltage - LED forward voltage) / desired current
R = (3.3V - 2.0V) / 0.01A = 130Ω minimum
470Ω is a safe, common choice
```

## RGB LED

### Common cathode vs common anode
- **Common cathode** — longest pin is GND; drive other pins HIGH to turn on
- **Common anode** — longest pin is 3.3V; drive other pins LOW to turn on

To test: connect longest pin to GND, touch others through a resistor to 3.3V.  
If LEDs light up → common cathode.

### Wiring (common cathode)
- Longest pin (cathode) → GND (pin 6)
- Red leg → 470Ω → GPIO 17 (pin 11)
- Green leg → 470Ω → GPIO 27 (pin 13)
- Blue leg → 470Ω → GPIO 22 (pin 15)

### PWM colour mixing
PWM (Pulse Width Modulation) rapidly switches a pin on/off. The duty cycle (0.0–1.0) controls brightness. Mix RGB channels to make any colour.

Uses `gpiozero.PWMLED` — see `modulate.py`.

## Button input with gpiozero

Wired a button to GPIO 2 and used `gpiozero.Button` with event callbacks to cycle an RGB LED through colours on press and turn it off on release.

### Key findings

- `gpiozero.Button` uses `when_pressed` and `when_released` callbacks — the script stays alive with `signal.pause()` rather than a polling loop.
- `RGBLED.color` expects a `colorzero.Color` object or an `(r, g, b)` tuple of floats `0.0–1.0`. Passing a plain string like `"red"` raises a `TypeError` at runtime.
- Import multiple classes from the same package in one line: `from gpiozero import Button, RGBLED`

### Ctrl+C not stopping the script when run via VS Code task

The "Deploy and Run" task SSHs without a PTY, so Ctrl+C kills the local SSH client but leaves Python running on the Pi. The fix is `-t` in the SSH command to allocate a pseudo-terminal, which forwards signals correctly:

```bash
ssh -t "$PI_USER@$PI_HOST" "python3 $REMOTE_PATH"
```

Without this, the GPIO pins stay claimed and subsequent scripts fail with "GPIO busy" — requiring `stop-all.py` to manually release them.

## Clearing busy GPIO pins

If a script exits without releasing GPIO and a subsequent script errors with "GPIO busy":

```bash
# Find the process using GPIO
sudo fuser /dev/gpiochip0

# Kill it
sudo kill <PID>
```

`gpiozero` releases GPIO automatically on clean exit. Using `try/finally` in scripts ensures cleanup even on error:

```python
try:
    # your code
except KeyboardInterrupt:
    pass
finally:
    led.off()
```

## Locale fix

The warning `setlocale: LC_ALL: cannot change locale (en_US.UTF-8)` means the locale hasn't been generated.

```bash
# Enable en_US.UTF-8 in locale.gen
sudo sed -i "s/# en_US.UTF-8 UTF-8/en_US.UTF-8 UTF-8/" /etc/locale.gen

# Generate it
sudo locale-gen

# Set as default
sudo update-locale LANG=en_US.UTF-8 LC_ALL=en_US.UTF-8

# Reboot to apply
sudo reboot
```

## Bash history — removing sensitive data

If a password was typed in a command (e.g. `nmcli ... password "hunter2"`), remove it from history:

```bash
# Remove specific lines matching a pattern
sed -i '/nmcli dev wifi connect/d' ~/.bash_history

# Or clear entire history
history -c && > ~/.bash_history
```

Note: `LC_ALL` set in the shell environment persists into SSH sessions — avoid setting credentials in environment variables or commands where possible.
